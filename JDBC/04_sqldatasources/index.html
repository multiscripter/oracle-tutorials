Подключение с объектом DataSource

В этом разделе рассматриваются объекты DataSource, которые являются предпочтительным средством получения соединения с источником данных. В дополнение к другим преимуществам, которые будут объяснены позже, объекты DataSource могут обеспечивать объединение пулов и распределенные транзакции. Эта функциональность необходима для работы с корпоративными базами данных. В частности, это является неотъемлемой частью технологии Enterprise JavaBeans (EJB).

В этом разделе показано, как получить соединение с помощью интерфейса DataSource и как использовать распределенные транзакции и объединение пулов. Оба они включают очень мало изменений кода в вашем приложении JDBC.

Работа, выполняемая для развертывания классов, делающих эти операции возможными, которую обычно делает системный администратор с помощью инструмента (например, Apache Tomcat или Oracle WebLogic Server), зависит от типа объекта DataSource, который развертывается. В результате большая часть этого раздела посвящена тому, как системный администратор устанавливает среду, чтобы программисты могли использовать объект DataSource для получения соединений.

Рассмотрены следующие темы:

  • Использование объектов DataSource для подключения
  • Развертывание основных объектов DataSource
  • Развертывание других реализаций DataSource
  • Получение и использование объединенных подключений (pool)
  • Развертывание распределенных транзакций
  • Использование подключений для распределенных транзакций

Использование объектов DataSource для получения соединения

В разделе Установке соединения вы узнали, как получить соединение, используя класс DriverManager. В этом разделе показано, как использовать объект DataSource для подключения к источнику данных, что является предпочтительным способом.

Объекты, созданные экземплярами классов, реализующих DataSource, представляют собой определенную СУБД или какой-либо другой источник данных, такой как файл. Если компания использует более одного источника данных, она будет развертывать отдельный объект DataSource для каждого из них. Интерфейс DataSource реализуется поставщиком драйверов. Его можно реализовать тремя способами:

  • Основная реализация DataSource создает стандартные объекты Connection, которые не объединены или не используются в распределенной транзакции.
  • Реализация DataSource, поддерживающая объединение пулов, создает объекты Connection, которые участвуют в пуле соединений, то есть соединения, которые могут быть переработаны.
  • Реализация DataSource, поддерживающая распределенные транзакции, создает объекты Connection, которые могут использоваться в распределенной транзакции, то есть транзакция, которая обращается к двум или более серверам СУБД.

Драйвер JDBC должен включать по крайней мере базовую реализацию DataSource. Например, драйвер Java JDBC включает реализацию org.apache.derby.jdbc.ClientDataSource, а для MySQL com.mysql.jdbc.jdbc2.optional.MysqlDataSource. Если ваш клиент работает на Java 8 compact profile 2, то Java DB JDBC-драйвер - это org.apache.derby.jdbc.BasicClientDataSource40. Для этого примера требуется compact profile 3 или выше.

Класс DataSource, который поддерживает распределенные транзакции, обычно также поддерживает поддержку пула соединений. Например, класс DataSource, предоставляемый поставщиком EJB, почти всегда поддерживает как пул соединений, так и распределенные транзакции.

Предположим, что владелец процветающей сети магазинов The Coffee Break, по сравнению с предыдущими примерами, решил еще больше расширить продажи кофе через Интернет. Учитывая, что ожидается большой объем онлайн-бизнеса, владельцу обязательно понадобится объединение пулов. Открытие и закрытие соединений связано с большими накладными расходами, и владелец ожидает, что эта система онлайн-заказов потребует значительного количества запросов и обновлений. Пул соединений можно использовать снова и снова, избегая затрат на создание нового соединения для каждого доступа к базе данных. Кроме того, у владельца теперь есть вторая СУБД, которая содержит данные для недавно приобретенной компании по обжариванию кофе. Это означает, что владелец захочет писать распределенные транзакции, которые используют как старый сервер СУБД, так и новый.

Владелец цепочки переконфигурировал компьютерную систему, чтобы обслуживать новую, большую клиентскую базу. Владелец купил самый последний драйвер JDBC и сервер приложений EJB, который работает с ним, чтобы иметь возможность использовать распределенные транзакции и получать повышенную производительность, связанную с пулом соединений. Доступны многие драйверы JDBC, совместимые с недавно приобретенным EJB-сервером. Теперь у владельца есть трехуровневая архитектура (three-tier architecture) с новым сервером приложений EJB и драйвером JDBC на среднем уровне (second tier) и двумя серверами СУБД в качестве третьего уровня (third tier). Клиентские компьютеры, делающие запросы, являются первым уровнем (first tier).

Развертывание основных объектов DataSource

Системному администратору необходимо развернуть объекты DataSource, чтобы команда программирования Coffee Break начала использовать их. Развертывание объекта DataSource состоит из трех задач:

  1. Создание экземпляра класса DataSource
  2. Настройка его свойств
  3. Регистрация его с помощью службы именования, которая использует API именования и интерфейса Java (JNDI)

Во-первых, рассмотрим самый простой случай, который заключается в использовании базовой реализации интерфейса DataSource, то есть той, которая не поддерживает объединение пулов или распределенных транзакций. В этом случае требуется только один объект DataSource, который необходимо развернуть. Базовая реализация DataSource создает те же типы соединений, что и класс DriverManager.

Создание экземпляра класса DataSource и настройка его свойств

Предположим, что компания, которая хочет только базовую реализацию DataSource, купила драйвер у поставщика JDBC DB Access, Inc. Этот драйвер включает в себя класс com.dbaccess.BasicDataSource, который реализует интерфейс DataSource. Следующий фрагмент кода создает экземпляр класса BasicDataSource и устанавливает его свойства. После развертывания экземпляра BasicDataSource программист может вызвать метод DataSource.getConnection(), чтобы получить соединение с базой данных компании CUSTOMER_ACCOUNTS. Во-первых, системный администратор создает объект ds базы данных DataSource с помощью конструктора по умолчанию. Затем системный администратор устанавливает три свойства. Обратите внимание, что следующий код обычно выполняется с помощью инструмента развертывания: 

com.dbaccess.BasicDataSource ds = new com.dbaccess.BasicDataSource();
ds.setServerName("grinder");
ds.setDatabaseName("CUSTOMER_ACCOUNTS");
ds.setDescription("Customer accounts database for billing");

Теперь переменная ds представляет собой базу данных CUSTOMER_ACCOUNTS, установленную на сервере. Любое соединение, созданное объектом базы данных BasicDataSource, будет соединением с базой данных CUSTOMER_ACCOUNTS.

Вторая строка ассоциирует или связывает BasicDataSource объект ds с логическим именем jdbc/billingDB. В следующем отрывке кода вы даете службе именования это логическое имя и возвращает объект BasicDataSource. Логическим именем может быть любая строка. Имя jdbc/billingDB похоже на имя пути, где последний элемент в пути аналогичен имени файла. В этом случае billingDB является логическим именем, которое присваивается BasicDataSource объекту ds. Подконтекст jdbc зарезервирован для привязки логических имен к объектам DataSource, поэтому jdbc всегда будет первой частью логического имени для источника данных.

Использование развернутого (deployed) объекта DataSource

После того как базовая реализация DataSource будет развернута системным администратором, она готова для использования программистом. Это означает, что программист может предоставить имя источника логических данных, привязанное к экземпляру класса DataSource, а служба именования JNDI вернет экземпляр этого класса DataSource. Затем метод getConnection() можно вызвать у этого объекта DataSource, чтобы получить соединение с источником данных, который он представляет. Например, программист может написать следующие две строки кода, чтобы получить объект DataSource, который создает соединение с базой данных CUSTOMER_ACCOUNTS. 

Context ctx = new InitialContext();
DataSource ds = (DataSource)ctx.lookup("jdbc/billingDB");

Первая строка кода получает исходный контекст в качестве отправной точки для извлечения объекта DataSource. Когда вы отправляете логическое имя jdbc/billingDB в метод lookup(). Метод возвращает объект DataSource, связанный системным администратором с jdbc/billingDB во время развертывания. Поскольку возвращаемое значение метода lookup() является объекти типа Object мы должны передать его более конкретному типу DataSource, прежде чем назначать его переменной ds.

Переменная ds является экземпляром класса com.dbaccess.BasicDataSource, который реализует интерфейс DataSource. Вызов метода ds.getConnection создает соединение с базой данных CUSTOMER_ACCOUNTS. 

Connection con = ds.getConnection("fernanda","brewed");

Метод getConnection() требует только имя пользователя и пароль, потому что переменная ds содержит остальную информацию, необходимую для установления соединения с базой данных CUSTOMER_ACCOUNTS, такой как имя и местоположение базы данных, в ее свойствах.

Преимущества объектов DataSource

Из-за его свойств объект DataSource является лучшей альтернативой, чем класс DriverManager для получения соединения. Программистам больше не нужно жестко кодировать имя драйвера или URL-адрес JDBC в своих приложениях, что делает их более переносимыми. Кроме того, свойства DataSource делают код обслуживания намного проще. Если произошли изменения, системный администратор может обновлять свойства источника данных и не беспокоиться об изменении каждого приложения, которое делает подключение к источнику данных. Например, если источник данных был перемещен на другой сервер, все системные администраторы должны были бы установить свойство serverName для нового имени сервера.

Помимо переносимости и простоты обслуживания, использование объекта DataSource для получения соединений может предложить другие преимущества. Когда интерфейс DataSource реализован для работы с реализацией ConnectionPoolDataSource, все соединения, создаваемые экземплярами этого класса DataSource, будут автоматически объединены в соединения. Аналогично, когда реализация DataSource реализована для работы с классом XADataSource, все соединения, которые он создает, автоматически будут соединениями, которые могут использоваться в распределенной транзакции. В следующем разделе показано, как развернуть эти типы реализаций DataSource.

Развертывание других реализаций DataSource

Системный администратор или другое лицо, работающее в таком качестве, может развернуть объект DataSource, чтобы соединения, которые он создает, объединяли соединения. Для этого он сначала развертывает объект ConnectionPoolDataSource, а затем развертывает объект DataSource, реализованный для работы с ним. Свойства объекта ConnectionPoolDataSource устанавливаются так, чтобы он представлял источник данных, к которому будут производиться соединения. После того, как объект ConnectionPoolDataSource зарегистрирован в службе именования JNDI, объект DataSource развернут. Обычно для объекта DataSource должны быть установлены только два свойства: description и dataSourceName. Значение, указанное для свойства dataSourceName, является логическим именем, идентифицирующим ранее запущенный объект ConnectionPoolDataSource, который является объектом, содержащим свойства, необходимые для соединения.

При развертывании объектов ConnectionPoolDataSource и DataSource вы можете вызвать метод DataSource.getConnection() у объекта DataSource и получить объединенное соединение. Это соединение будет связано с источником данных, указанным в свойствах объекта ConnectionPoolDataSource.

В следующем примере описывается, как системный администратор The Coffee Break будет развертывать объект DataSource, реализованный для обеспечения объединенных соединений. Системный администратор обычно использует инструмент развертывания, поэтому фрагменты кода, показанные в этом разделе, представляют собой код, который будет выполнять инструмент развертывания.

Чтобы получить лучшую производительность, компания Coffee Break купила драйвер JDBC от DB Access, Inc., который включает класс com.dbaccess.ConnectionPoolDS, который реализует интерфейс ConnectionPoolDataSource. Системный администратор создает экземпляр этого класса, устанавливает его свойства и регистрирует его с помощью службы именования JNDI. Coffee Break купила свой DataSource-класс com.applogic.PooledDataSource, от своего поставщика EJB-сервера, Application Logic, Inc. Класс com.applogic.PooledDataSource реализует пул соединений ConnectionPoolDataSource, используя базовую поддержку, предоставляемую классом com.dbaccess.ConnectionPoolDS.

Сначала необходимо установить объект ConnectionPoolDataSource. Следующий код создает экземпляр com.dbaccess.ConnectionPoolDS и устанавливает его свойства: 

com.dbaccess.ConnectionPoolDS cpds = new com.dbaccess.ConnectionPoolDS();
cpds.setServerName("creamer");
cpds.setDatabaseName("COFFEEBREAK");
cpds.setPortNumber(9040);
cpds.setDescription("Connection pooling for " + "COFFEEBREAK DBMS");

После развертывания объекта ConnectionPoolDataSource системный администратор развертывает объект DataSource. Следующий код регистрирует объект cpds класса com.dbaccess.ConnectionPoolDS с помощью службы именования JNDI. Обратите внимание, что логическое имя, связанное с переменной cpds, имеет пул субконтексов, добавленный под подконтекстом jdbc, который похож на добавление подкаталога в другой подкаталог в иерархической файловой системе. Логическое имя любого экземпляра класса com.dbaccess.ConnectionPoolDS всегда будет начинаться с jdbc/pool. Oracle рекомендует помещать все объекты ConnectionPoolDataSource в подконтекст jdbc/pool: 

Context ctx = new InitialContext();
ctx.bind("jdbc/pool/fastCoffeeDB", cpds);

Затем развертывается класс DataSource, который реализуется для взаимодействия с переменной cpds и другими экземплярами класса com.dbaccess.ConnectionPoolDS. Следующий код создает экземпляр этого класса и устанавливает его свойства. Обратите внимание, что для этого экземпляра com.applogic.PooledDataSource задано только два свойства. Свойство description установлено, потому что оно всегда требуется. Другое свойство, которое установлено, dataSourceName, дает логическое имя JNDI для cpds, которое является экземпляром класса com.dbaccess.ConnectionPoolDS. Другими словами, cpds представляет объект ConnectionPoolDataSource, который будет реализовывать пул соединений для объекта DataSource.

Следующий код, который, вероятно, будет выполняться инструментом развертывания, создает объект PooledDataSource, устанавливает его свойства и связывает его с логическим именем jdbc/fastCoffeeDB: 

com.applogic.PooledDataSource ds = new com.applogic.PooledDataSource();
ds.setDescription("produces pooled connections to COFFEEBREAK");
ds.setDataSourceName("jdbc/pool/fastCoffeeDB");
Context ctx = new InitialContext();
ctx.bind("jdbc/fastCoffeeDB", ds);

В этом фрагменте кода развертывается объект DataSource, из которого приложение может получать объединенные соединения с базой данных COFFEEBREAK.

Получение и использование объединенных подключений (pool)

Пул соединений - это кеш объектов подключения к базе данных. Объекты представляют физические подключения к базе данных, которые могут использоваться приложением для подключения к базе данных. Во время выполнения приложение запрашивает соединение из пула. Если пул содержит соединение, которое может удовлетворить запрос, оно возвращает соединение. Если соединения не найдены, создается новое соединение и возвращается в приложение. Приложение использует соединение для выполнения некоторой работы с базой данных, а затем возвращает соединение обратно в пул. Затем объект соединения доступен для следующего запроса на соединение с БД.

Пулы соединений способствуют повторному использованию объектов подключения и уменьшают количество создаваемых объектов соединения. Пулы соединений значительно повышают производительность приложений с интенсивным использованием баз данных, поскольку создание объектов соединения является дорогостоящим как с точки зрения времени, так и с точки зрения ресурсов.

Теперь, когда эти объекты DataSource и ConnectionPoolDataSource развернуты, программист может использовать объект DataSource для получения объединенного соединения. Код для получения объединенного соединения аналогичен коду для получения непустого соединения, как показано в следующих двух строках: 

ctx = new InitialContext();
ds = (DataSource)ctx.lookup("jdbc/fastCoffeeDB");

Переменная ds представляет объект DataSource, который создает объединенные соединения с базой данных COFFEEBREAK. Вам нужно получить этот объект DataSource только один раз, потому что вы можете использовать его для создания как можно большего количества объединенных соединений. Вызов метода getConnection() на переменной ds автоматически создает объединенное соединение, потому что объект DataSource, который представляет переменная ds, настроен для создания объединенных соединений.

Пул соединений обычно прозрачен для программиста. Вам нужно всего две вещи, когда вы используете объединенные соединения:

  1. Для получения соединения используйте объект DataSource, а не класс DriverManager. В следующей строке кода ds - это объект DataSource, реализованный и развернутый, так что он будет создавать пустые соединения, а имя пользователя и пароль - это переменные, которые представляют учетные данные пользователя, имеющего доступ к базе данных: 

    Connection con = ds.getConnection(username, password);

  2. Используйте оператор finally для закрытия объединенного соединения. Следующий блок finally появится после блока try/catch, который применяется к коду, в котором использовалось объединенное соединение: 

    try {
    Connection con = ds.getConnection(username, password);
    // ... code to use the pooled
    // connection con
} catch (Exception ex {
    // ... code to handle exceptions
} finally {
    if (con != null) con.close();
}

В противном случае приложение, использующее объединенное соединение, идентично приложению, использующему обычное соединение. Единственное, что может заметить прикладной программист, когда происходит объединение пулов, - это то, что производительность лучше.

Следующий пример кода получает объект DataSource, который создает соединения с базой данных COFFEEBREAK и использует его для обновления цены в таблице COFFEES: 

import java.sql.*;
import javax.sql.*;
import javax.ejb.*;
import javax.naming.*;
public class ConnectionPoolingBean implements SessionBean {
    // ...
    public void ejbCreate() throws CreateException {
        ctx = new InitialContext();
        ds = (DataSource)ctx.lookup("jdbc/fastCoffeeDB");
    }
    public void updatePrice(float price, String cofName,
                            String username, String password)
        throws SQLException{
        Connection con;
        PreparedStatement pstmt;
        try {
            con = ds.getConnection(username, password);
            con.setAutoCommit(false);
            pstmt = con.prepareStatement("UPDATE COFFEES " +
                        "SET PRICE = ? " +
                        "WHERE COF_NAME = ?");
            pstmt.setFloat(1, price);
            pstmt.setString(2, cofName);
            pstmt.executeUpdate();
            con.commit();
            pstmt.close();
        } finally {
            if (con != null) con.close();
        }
    }
    private DataSource ds = null;
    private Context ctx = null;
}

Соединение в этом примере кода участвует в пуле соединений, потому что верно следующее:

  • Развернут экземпляр класса, реализующего ConnectionPoolDataSource.
  • Развернут экземпляр класса, реализующего DataSource, а значение, установленное для его свойства dataSourceName, является логическим именем, которое было привязано к ранее развернутому ConnectionPoolDataSource.

Обратите внимание: хотя этот код очень похож на код, который вы видели ранее, он отличается следующим образом:

  • Он импортирует пакеты javax.sql, javax.ejb и javax.naming в дополнение к java.sql

    Интерфейсы DataSource и ConnectionPoolDataSource находятся в пакете javax.sql и конструкторе JNDI InitialContext и методе Context.lookup() являются частью пакета javax.naming. Этот конкретный пример кода представлен в виде компонента EJB, который использует API из пакета javax.ejb. Цель этого примера - показать, что вы используете объединенное соединение так же, как вы используете несвязанное соединение, поэтому вам не нужно беспокоиться о понимании EJB API.

  • Он использует объект DataSource, чтобы получить соединение вместо использования средства DriverManager.

  • Он использует блок finally, чтобы убедиться, что соединение закрыто.

Получение и использование объединенного соединения похоже на получение и использование обычного соединения. Когда кто-то, действуя как системный администратор, правильно развернул объект ConnectionPoolDataSource и объект DataSource, приложение использует этот объект DataSource для получения объединенного соединения. Однако приложение должно использовать блок finally для закрытия объединенного соединения. Для простоты в предыдущем примере использовался блок finally, но не блок catch. Если исключение выбрано методом в блоке try, оно будет выбрано по умолчанию, и предложение finally будет выполнено в любом случае.

Развертывание распределенных транзакций

Объекты DataSource могут быть развернуты для получения соединений, которые могут использоваться в распределенных транзакциях. Как и при объединении пулов, необходимо развернуть два разных экземпляра класса: объект XADataSource и объект DataSource, который используется для работы с ним.

Предположим, что EJB-сервер, купленный покупателем The Coffee Break, включает в себя класс DataSource из пакета com.applogic.TransactionalDS, который работает с классом XADataSource, таким как com.dbaccess.XATransactionalDS. Тот факт, что он работает с любым классом XADataSource, делает EJB-сервер переносимым драйверами JDBC. Когда объекты DataSource и XADataSource развернуты, созданные соединения смогут участвовать в распределенных транзакциях. В этом случае класс com.applogic.TransactionalDS реализован так, чтобы создаваемые соединения также объединялись в соединения, что обычно будет иметь место для классов DataSource, предоставляемых как часть реализации EJB-сервера.

Сначала должен быть развернут объект XADataSource. Следующий код создает экземпляр com.dbaccess.XATransactionalDS и устанавливает его свойства: 

com.dbaccess.XATransactionalDS xads = new com.dbaccess.XATransactionalDS();
xads.setServerName("creamer");
xads.setDatabaseName("COFFEEBREAK");
xads.setPortNumber(9040);
xads.setDescription("Distributed transactions for COFFEEBREAK DBMS");

Следующий код регистрирует xads типа com.dbaccess.XATransactionalDS с помощью службы именования JNDI. Обратите внимание, что логическое имя, связанное с xads, имеет подконтекст xa, добавленный в jdbc. Oracle рекомендует, чтобы логическое имя любого экземпляра класса com.dbaccess.XATransactionalDS всегда начиналось с jdbc/xa. 

Context ctx = new InitialContext();
ctx.bind("jdbc/xa/distCoffeeDB", xads);

Затем развертывается объект DataSource, который реализуется для взаимодействия с xads и другими объектами XADataSource. Обратите внимание, что класс DataSource из пакета com.dbaccess.XATransactionalDS, может работать с классом XADataSource от любого поставщика драйверов JDBC. Развертывание объекта DataSource включает в себя создание экземпляра класса com.dbaccess.XATransactionalDS и настройку его свойств. Свойству DataSource присваивается значение jdbc/xa/distCoffeeDB, логическое имя, связанное с com.dbaccess.XATransactionalDS. Это класс XADataSource, который реализует возможности распределенных транзакций для класса DataSource. Следующий код развертывает экземпляр класса DataSource: 

com.applogic.TransactionalDS ds = new com.applogic.TransactionalDS();
ds.setDescription("Produces distributed transaction " +
                  "connections to COFFEEBREAK");
ds.setDataSourceName("jdbc/xa/distCoffeeDB");
Context ctx = new InitialContext();
ctx.bind("jdbc/distCoffeeDB", ds);

Теперь, когда развертываются экземпляры классов com.applogic.TransactionalDS и com.dbaccess.XATransactionalDS, приложение может вызвать метод getConnection() для экземпляров класса TransactionalDS для подключения к базе данных COFFEEBREAK, которая может использоваться в распределенных транзакциях.

Using Connections for Distributed Transactions

Чтобы получить соединение, которое может использоваться для распределенных транзакций, необходимо использовать объект DataSource, который был правильно реализован и развернут, как показано в разделе Развертывание распределенных транзакций. С таким объектом DataSource вызовите метод getConnection на нем. После того, как у вас есть соединение, используйте его так же, как и любое другое соединение. Поскольку jdbc/distCoffeesDB связан с объектом XADataSource в службе именования JNDI, следующий код создает объект Connection, который может использоваться в распределенных транзакциях: 

Context ctx = new InitialContext();
DataSource ds = (DataSource)ctx.lookup("jdbc/distCoffeesDB");
Connection con = ds.getConnection();

Существуют некоторые незначительные, но важные ограничения на то, как это соединение используется, когда оно является частью распределенной транзакции. Менеджер транзакций контролирует, когда начинается распределенная транзакция и когда она совершена или откат; поэтому код приложения никогда не должен вызывать методы Connection.commit() или Connection.rollback(). Приложение также не должно ссылаться на Connection.setAutoCommit(true), что позволяет использовать режим автоматической фиксации, поскольку это также будет мешать управлению транзакционным менеджером границ транзакций. Это объясняет, почему новое соединение, созданное в рамках распределенной транзакции, отключено по умолчанию для режима автоматической фиксации. Обратите внимание, что эти ограничения применяются только тогда, когда соединение участвует в распределенной транзакции; нет никаких ограничений, пока соединение не является частью распределенной транзакции.

В следующем примере предположим, что был отправлен заказ кофе, который запускает обновления для двух таблиц, которые находятся на разных серверах СУБД. Первая таблица представляет собой новую таблицу INVENTORY, а вторая - таблицу COFFEES. Поскольку эти таблицы находятся на разных серверах СУБД, транзакция, включающая их оба, будет распределенной транзакцией. Код в следующем примере, который получает соединение, обновляет таблицу COFFEES и закрывает соединение, является второй частью распределенной транзакции.

Обратите внимание, что код явно не фиксирует или откатывает обновления, поскольку область распределенной транзакции контролируется базовой системной инфраструктурой среднего уровня. Кроме того, предполагая, что соединение, используемое для распределенной транзакции, представляет собой объединенное соединение, приложение использует блок finally для закрытия соединения. Это гарантирует, что действительное соединение будет закрыто, даже если выбрано исключение, тем самым гарантируя, что соединение будет возвращено в пул соединений, подлежащих переработке.

Следующий пример кода иллюстрирует бизнес-компонент, который является классом, который реализует методы, которые могут быть вызваны клиентским компьютером. Цель этого примера - продемонстрировать, что код приложения для распределенной транзакции ничем не отличается от другого кода, за исключением того, что он не вызывает методы commit(), rollback() и setAutoCommit(true) интерфейсаConnection. Поэтому вам не нужно беспокоиться о понимании используемого API EJB. 

import java.sql.*;
import javax.sql.*;
import javax.ejb.*;
import javax.naming.*;
public class DistributedTransactionBean implements SessionBean {
    // ...
    public void ejbCreate() throws CreateException {
        ctx = new InitialContext();
        ds = (DataSource)ctx.lookup("jdbc/distCoffeesDB");
    }
    public void updateTotal(int incr, String cofName, String username,
                            String password)
        throws SQLException {
        Connection con;
        PreparedStatement pstmt;
        try {
            con = ds.getConnection(username, password);
            pstmt = con.prepareStatement("UPDATE COFFEES " +
                        "SET TOTAL = TOTAL + ? " +
                        "WHERE COF_NAME = ?");
            pstmt.setInt(1, incr);
            pstmt.setString(2, cofName);
            pstmt.executeUpdate();
            stmt.close();
        } finally {
            if (con != null) con.close();
        }
    }
    private DataSource ds = null;
    private Context ctx = null;
}
